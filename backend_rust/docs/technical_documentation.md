# Техническая документация: Errant Fox Backend

## 1. Обзор системы
Errant Fox — это серверное приложение на языке Rust, предназначенное для управления медиа-файлами (видео) и проведения глубокой аналитики фехтовальных поединков (HEMA). Система обеспечивает хранение видео, транскодирование, управление комментариями с привязкой к таймкодам и ведение судейских протоколов.

### Ключевые технологии:
*   **Язык**: Rust (Edition 2021)
*   **Веб-сервер**: Warp (WebSockets + HTTP)
*   **База данных**: SQLite + Diesel (ORM)
*   **Сериализация**: Protocol Buffers (Protobuf) через библиотеку Tonic/Prost
*   **Аутентификация**: JWT (JSON Web Tokens) + Bcrypt для хэширования паролей

---

## 2. Архитектура

Система построена на принципах реактивного взаимодействия через WebSockets.

1.  **Communication Layer**: Всё взаимодействие между фронтендом (Noodl) и бэкендом происходит через единое WebSocket-соединение. Команды типизированы с помощью Protobuf (`ClientToServerCmd`), что минимизирует ошибки парсинга.
2.  **Database Layer**: SQLite используется для хранения всех метаданных. Diesel обеспечивает типобезопасность запросов.
3.  **Storage Layer**: Исходные видео и оптимизированные для веба копии (MP4) хранятся в структурированной файловой системе.
4.  **Business Logic Layer**: Обработчики команд (`ws_handers.rs`) управляют состоянием сессий, правами доступа и манипуляциями с данными.

## 4. Схема базы данных

После рефакторинга система использует упрощенную консолидированную схему:

### Таблица `users` (Пользователи и Бойцы)
Единая таблица для администраторов системы и участников боев.
*   `id` (TEXT, PK): Уникальный строковый идентификатор.
*   `login` (TEXT): Логин для входа.
*   `name` (TEXT): Отображаемое имя.
*   `password_hash` (TEXT, nullable): Хэш пароля (только для пользователей с доступом).
*   `is_admin` (BOOLEAN): Флаг прав администратора.
*   `color` (TEXT): Индивидуальный цвет бойца (для UI).
*   `avatar` (TEXT, nullable): Ссылка на аватар или Base64.

### Таблица `media_files` (Видеофайлы)
*   `id` (TEXT, PK): Хэш файла или UUID.
*   `title` (TEXT): Название видео.
*   `fight_date` (DATETIME, nullable): Дата проведения боя.
*   `participant_a_id` / `participant_b_id` (TEXT): Ссылки на `users.id`.
*   `total_score_a` / `total_score_b` (INTEGER): Суммарный счет.

### Таблица `hema_bouts` (Сходы/Обмены)
*   `id` (INTEGER, PK): Автоинкремент.
*   `video_hash` (TEXT): Ссылка на видеофайл.
*   `start_time` / `end_time` (FLOAT): Таймкоды начала и конца обмена (в секундах).
*   `participant_a_id` / `participant_b_id` (TEXT): Кто участвовал в обмене.
*   `score_a` / `score_b` (INTEGER): Очки за обмен.
*   `move_a_id` / `move_b_id` (INTEGER): Использованный прием (ссылка на `hema_moves`).
*   `hit_zone_a` / `hit_zone_b` (TEXT): Зона попадания.

## 5. API Reference (WebSocket команды)

Команды отправляются как обертка `ClientToServerCmd` в формате Protobuf.

### Управление сессией
*   `Login`: Аутентификация. Возвращает JWT токен.
*   `Logout`: Завершение сессии.

### Работа с контентом
*   `OpenMediaFile`: Загрузка данных конкретного видео (ссылки на потоки, комментарии, сходы).
*   `RenameMediaFile` / `DelMediaFile`: Манипуляция файлами.
*   `AddComment` / `EditComment` / `DelComment`: Текстовые заметки и графические рисунки на таймлайне.

### HEMA Аналитика
*   `HemaListUsers`: Получение списка всех доступных бойцов/пользователей.
*   `HemaAddUser` / `HemaUpdateUser`: Создание и редактирование профилей участников.
*   `HemaAddBout` / `HemaUpdateBout` / `HemaDelBout`: Управление судейским протоколом видео.

## 6. Примечания для разработчика

1.  **Синхронизация**: Сервер автоматически рассылает обновления (`ServerToClientCmd`) всем пользователям, просматривающим тот же файл.
2.  **Безопасность**: Все команды, кроме `Login`, требуют наличия валидного JWT токена в заголовках WebSocket-запроса (или первого сообщения).
3.  **Транскодирование**: Если видео загружено в неоптимальном формате, сервер автоматически создаст `video.mp4` в папке файла.
